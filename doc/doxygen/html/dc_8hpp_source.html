130 /**
131  * \ingroup rpc
132  * \brief The distributed control object is primary means of communication
133  * between the distributed GraphLab processes.
134  *
135  * The distributed_control object provides asynchronous, multi-threaded
136  * Remote Procedure Call (RPC) services to allow distributed GraphLab
137  * processes to communicate with each other. Currently, the only
138  * communication method implemented is TCP/IP.
139  * There are several ways of setting up the communication layer, but the most
140  * reliable, and the preferred method, is to "bootstrap" using MPI. See your
141  * local MPI documentation for details on how to launch MPI jobs.
142  *
143  * To construct a distributed_control object, the simplest method is to just
144  * invoke the default constructor.
145  *
146  * \code
147  * // initialize MPI
148  * mpi_tools::init(argc, argv);
149  * // construct distributed control object
150  * graphlab::distributed_control dc;
151  * \endcode
152  *
153  * After which all distributed control services will operate correctly.
154  *
155  * Each process is assigned a sequential process ID at starting at 0.
156  * i.e. The first process will have a process ID of 0, the second process
157  * will have an ID of 1, etc. distributed_control::procid() can be used to
158  * obtain the current machine's process ID, and distributed_control::numprocs()
159  * can be used to obtain the total number of processes.
160  *
161  * The primary functions used to communicate between processes are
162  * distributed_control::remote_call() and
163  * distributed_control::remote_request(). These functions are thread-safe and
164  * can be called very rapidly as they only write into a local buffer.
165  * Communication is handled by a background thread. On the remote side,
166  * RPC calls are handled in parallel by a thread pool, and thus may be
167  * parallelized arbitrarily. Operations such as
168  * distributed_control::full_barrier(), or the sequentialization key
169  * can be used to get finer grained control over order of execution on the
170  * remote machine.
171  *
172  * A few other additional helper functions are also provided to support
173  * "synchronous" modes of communication. These functions are not thread-safe
174  * and can only be called on one thread per machine. These functions block
175  * until all machines call the same function. For instance, if gather() is
176  * called on one machine, it will not return until all machines call gather().
177  *
178  * \li distributed_control::barrier()
179  * \li distributed_control::full_barrier()
180  * \li distributed_control::broadcast()
181  * \li distributed_control::all_reduce()
182  * \li distributed_control::all_reduce2()
183  * \li distributed_control::gather()
184  * \li distributed_control::all_gather()
185  *
186  * \note These synchronous operations are modeled after some MPI collective
187  * operations. However, these operations here are not particularly optimized
188  * and will generally be slower than their MPI counterparts. However, the
189  * implementations here are much easier to use, relying extensively on
190  * serialization to simplify communication.
191  *
192  * To support Object Oriented Programming like methodologies, we allow the
193  * creation of <b>Distributed Objects</b> through graphlab::dc_dist_object.
194  * dc_dist_object allows a class to construct its own local copy of
195  * a distributed_control object allowing instances of the class to communicate
196  * with each other across the network.
197  *
198  * See \ref RPC for usage examples.
199  */

547 /**
548  * \brief Performs a non-blocking RPC call to the target machine
549  * to run the provided function pointer.
550  *
551  * remote_call() calls the function "fn" on a target remote machine. Provided
552  * arguments are serialized and sent to the target.
553  * Therefore, all arguments are necessarily transmitted by value.
554  * If the target function has a return value, the return value is lost.
555  *
556  * remote_call() is non-blocking and does not wait for the target machine
557  * to complete execution of the function. Different remote_calls may be handled
558  * by different threads on the target machine and thus the target function
559  * should be made thread-safe.
560  * Alternatively, see set_sequentialization_key()
561  * to force sequentialization of groups of remote calls.
562  *
563  * If blocking operation is desired, remote_request() may be used.
564  * Alternatively, a full_barrier() may also be used to wait for completion of
565  * all incomplete RPC calls.
566  *
567  * Example:
568  * \code
569  * // A print function is defined
570  * void print(std::string s) {
571  * std::cout << s << "\n";
572  * }
573  *
574  * ... ...
575  * // call the print function on machine 1 to print "hello"
576  * dc.remote_call(1, print, "hello");
577  * \endcode
578  *
579  *
580  *
581  * \param targetmachine The ID of the machine to run the function on
582  * \param fn The function to run on the target machine
583  * \param ... The arguments to send to Fn. Arguments must be serializable.
584  * and must be castable to the target types.
585  */
586  void remote_call(procid_t targetmachine, Fn fn, ...);

590 /**
591  * \brief Performs a non-blocking RPC call to a collection of machines
592  * to run the provided function pointer.
593  *
594  * This function calls the provided function pointer on a collection of
595  * machines contained in the iterator range [begin, end).
596  * Provided arguments are serialized and sent to the target.
597  * Therefore, all arguments are necessarily transmitted by value.
598  * If the target function has a return value, the return value is lost.
599  *
600  * This function is functionally equivalent to:
601  *
602  * \code
603  * while(machine_begin != machine_end) {

1262  * This function is equivalent to all_reduce(), but with an externally
1263  * defined PlusEqual function.
1264  *
1265  * Each machine calls all_reduce() with a object which is serializable
1266  * and a function "plusequal" which combines two instances of the object.
1267  * When all_reduce2() returns, the "data"
1268  * variable will contain a value corresponding to adding up the objects
1269  * contributed by each machine using the plusequal function.
1270  *
1271  * Where U is the type of the object, the plusequal function must be of
1272  * the form:
1273  * \code
1274  * void plusequal(U& left, const U& right);
1275  * \endcode
1276  * and must implement the equivalent of <code>left += right; </code>
1277  *
1278  * Example:
1279  * \code
1280  * void int_plus_equal(int& a, const int& b) {
1281  * a+=b;
1282  * }
1283  *
1284  * int i = 1;
1285  * dc.all_reduce2(i, int_plus_equal);
1286  * // since each machine contributed the value "1",
1287  * // all machines will have i = numprocs() here.
1288  * \endcode
1289  *
1290  * \param data A piece of data to perform a reduction over.
1291  * \param plusequal A plusequal function on the data. Must have the prototype
1292  * void plusequal(U&, const U&)
1293  * \param control Optional parameter. Defaults to false. If set to true,
1294  * this will marked as control plane communication and will
1295  * not register in bytes_received() or bytes_sent(). This must
1296  * be the same on all machines.
1297  */
